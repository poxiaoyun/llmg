## Goal

Build a second-developed frontend UI that can coexist with the product's existing frontends, so the custom UI can be compared against the upstream default UI while `main` remains suitable for upstream synchronization.

The target custom frontend root is:

- `web/llmg/`

The existing frontend roots are baselines:

- `web/default/` is the current React 19 / Rsbuild / Base UI / Tailwind frontend and must remain the upstream comparison baseline.
- `web/classic/` is the existing React 18 / Vite / Semi Design classic frontend and must remain the legacy comparison baseline.

Do not redesign `web/default/` directly for custom UI work. Copy or port from it into `web/llmg/`, then keep custom UI changes inside `web/llmg/` unless the change is explicitly a shared integration or shared bug fix.

## Current Architecture Snapshot

Backend:

- Go API gateway using Gin, GORM, Redis/in-memory cache, JWT/session auth, OAuth, WebAuthn, relay provider adapters, billing, and admin APIs.
- Layering is `router/` -> `controller/` -> `service/` -> `model/`.
- JSON operations in business code must use `common/json.go` wrappers.
- DB code must remain compatible with SQLite, MySQL, and PostgreSQL.

Frontend:

- `web/default/` uses React 19, TypeScript, Rsbuild, TanStack Router, React Query, Zustand, Base UI, Tailwind CSS v4, shadcn-style local UI components, and i18next.
- `web/classic/` uses React 18, Vite, Semi Design, react-router-dom, and classic i18n files.
- Use Bun for frontend install/build/dev scripts.
- `web/default/src/routeTree.gen.ts` is generated by TanStack Router tooling.
- `web/default/dist` and `web/classic/dist` are ignored generated build outputs and were absent during the scan.

Runtime frontend selection:

- `main.go` embeds `web/default/dist` and `web/classic/dist`.
- `router/web-router.go` builds a theme-aware static filesystem and serves SPA fallbacks based on `common.GetTheme()`.
- `common/constants.go` currently accepts only `default` and `classic` in `SetTheme`.
- `setting/system_setting/theme.go` stores `theme.frontend`.
- `controller/option.go` validates `theme.frontend`.
- Admin UI selectors exist in `web/default/src/features/system-settings/general/system-info-section.tsx` and `web/classic/src/components/settings/OtherSetting.jsx`.

Important limitation:

- The current server selects one active frontend theme globally. Coexistence currently means multiple embedded/selectable frontends, not per-request side-by-side routing. For live side-by-side comparison during development, run multiple frontend dev servers on different ports against the same backend. If production side-by-side URLs are required later, add that as a separate routing change.

## Branch And Upstream Rules

- Keep `main` clean and tracking upstream/fork sync. Do not commit custom UI changes on `main`.
- Do custom work on a long-lived branch such as `codex-frontend-ui` or a task branch based on it.
- Before future Codex edits, run `git status --short --branch`. If on `main`, switch to or create a work branch first.
- To sync later, update `main` from the upstream source, then merge or rebase `main` into the custom UI branch. Resolve conflicts by preserving upstream behavior in `web/default/`, `web/classic/`, backend APIs, and shared settings unless the task explicitly changes shared behavior.
- If an actual upstream remote is needed, add it only after the upstream URL is confirmed.

## Modification Boundaries

Custom UI area:

- Prefer editing `web/llmg/**` for all second-developed UI screens, layouts, styles, assets, route composition, and frontend-only behavior.
- Keep `web/llmg` aligned with `web/default` API contracts unless a backend change is explicitly requested.
- Keep package management with Bun. Avoid new dependencies unless the existing stack cannot reasonably solve the requirement.

Comparison baselines:

- Avoid custom redesign edits in `web/default/**`.
- Avoid refactors in `web/classic/**`.
- Baseline directories may be touched only for additive shared integration needed to expose the custom theme, or for upstream-compatible bug fixes. Document such changes in the final response.

Allowed shared integration files for introducing or maintaining the third theme:

- `main.go`
- `router/web-router.go`
- `common/embed-file-system.go`
- `common/constants.go`
- `setting/system_setting/theme.go`
- `controller/option.go`
- `makefile`
- `Dockerfile`
- `Dockerfile.dev`
- `.github/workflows/**` only when release/build automation needs the new frontend
- `web/default/src/features/system-settings/**` only to expose/select the new theme
- `web/classic/src/components/settings/OtherSetting.jsx` only to expose/select the new theme

Protected and high-risk areas:

- Do not remove, rename, replace, or obscure protected project/author identifiers covered by `AGENTS.md`.
- Do not change Go module paths, import roots, package metadata, README/license attribution, Docker image identity, or CI metadata for branding-only reasons.
- Do not edit generated `dist` artifacts by hand.
- Do not edit `web/default/src/routeTree.gen.ts` by hand; let TanStack Router tooling regenerate it.
- Do not introduce backend raw SQL without cross-database handling.
- For billing expression work, read `pkg/billingexpr/expr.md` before editing.

## Target Third-Theme Implementation Map

When implementation begins, use this additive map:

1. Create `web/llmg/` from `web/default/` as the starting point unless the user requests a different base.
2. Keep the same API base behavior: same-origin `/api`, `/mj`, and `/pg` requests through the backend/dev proxy.
3. Add a third theme value, recommended as `llmg`, across backend validation, common theme selection, admin selectors, and i18n labels.
4. Extend embed/static serving to include `web/llmg/dist` without changing existing `default` and `classic` behavior.
5. Add `build-frontend-llmg` and `dev-web-llmg` style scripts in the root `makefile` only when `web/llmg` exists.
6. Update Docker and release workflows only when the custom frontend must be built into production artifacts.
7. Keep default and classic themes buildable after adding the third theme.

## Frontend Coding Rules

- Follow `web/default/AGENTS.md` for React/TypeScript style unless this index is stricter.
- Use functional components and typed props.
- Avoid `any`; prefer explicit types or `unknown`.
- Use project UI primitives and existing layout/data-table/form patterns before adding new abstractions.
- Keep custom UI text internationalized. The current default frontend uses flat locale JSON files and source-like keys; follow the actual existing i18n pattern and run the sync tooling when keys change.
- Add or update translations for `en`, `zh`, `fr`, `ru`, `ja`, and `vi` when user-facing text is added in shared/default/custom UI.
- Use TanStack Router file routes; do not manually patch generated route output.
- Keep API calls through the shared axios/React Query conventions.
- Keep styles in Tailwind/CSS variable patterns already used by the app.

## Verification Gates

For custom frontend-only changes:

- `cd web/llmg && bun run typecheck`
- `cd web/llmg && bun run build`

For changes touching `web/default`:

- `cd web/default && bun run typecheck`
- `cd web/default && bun run build`
- `cd web/default && bun run i18n:sync` when user-facing text changes

For changes touching `web/classic`:

- `cd web/classic && bun run build`

For shared Go/theme integration:

- Ensure all embedded frontend paths have `dist/index.html` available before compiling `main.go`; the repository ignores frontend `dist` outputs, so a clean checkout may need frontend builds or placeholder dist files for Go embed.
- Run targeted Go tests for touched packages when possible.
- Run broader `go test ./...` only after embed prerequisites are satisfied.

For visual UI work:

- Start the relevant dev server and inspect the changed route in browser automation or an equivalent local browser.
- Check at least a desktop and a mobile viewport for navigation, dialogs/drawers, tables, overflow, and text wrapping.
