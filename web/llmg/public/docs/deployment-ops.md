# 部署与运行维护

LLMG 的运行方式可以从最简单的 Docker Compose 启动，逐步扩展到容器化交付、本地开发和更复杂的外部依赖组合。无论你采用哪种方式，都建议把部署与配置、数据持久化和升级策略分开考虑。

## 推荐部署入口

### Docker Compose

对于大多数团队，Compose 仍然是最低成本的上线方式。它足够直观，也更方便一起管理数据库、缓存和代理层配置。

### Docker 直接运行

如果你已经有现成的调度环境，直接运行镜像会更灵活。但要提前规划持久化目录、环境变量和外部数据库连接。

### 本地开发

当前仓库已经验证过一套本地开发链路：后端跑在 `:3000`，`web/llmg` 前端开发服务跑在 `:3001`，并把接口代理回本地后端。

## 运行维护时要关注什么

- **配置文件和环境变量**：上线前明确数据库、Redis、支付、OAuth 与上游渠道配置来源。
- **持久化目录**：不要把生产数据留在容器临时层里。
- **前端构建产物**：如果改动了 `web/llmg`，发布前要重新构建前端资源。
- **版本升级**：升级前先确认数据库兼容性和是否有需要执行的迁移脚本。

## 本地调试建议

### 前后端分离调试

适合频繁做界面或交互迭代。你可以单独跑 `web/llmg`，同时复用本地或远程后端。

### 控制变量调试

当你在排查渠道、计费或认证问题时，尽量一次只改一个变量，例如只切换一个上游、只替换一把密钥，避免多变量叠加造成误判。

## 发布前清单

1. 前端 `typecheck` 与 `build` 均通过。
2. 关键调用路径至少完成一次手工验证。
3. 管理员账号、支付配置和第三方回调都已经确认。
4. 关键模型的价格展示、余额扣减和日志记录可对齐。

## 运行中的巡检建议

- 定期检查价格页、用量概览和调用日志是否一致。
- 关注上游渠道的认证状态和响应质量。
- 在重要升级前备份数据库与关键配置。
- 对外提供服务前，先确认默认账号、默认密码等初始化行为已经按你的环境要求处理完毕。
