# 渠道与路由

渠道是 LLMG 与上游供应商之间的连接单位。你可以通过渠道把 OpenAI、Claude、Gemini、Azure、Rerank 或其他兼容服务接入进来，再用分组、模型映射和调度策略决定请求最终走向哪里。

## 配置一个渠道时需要确认什么

- **渠道类型**：选择正确的供应商或协议格式。
- **接入地址**：如果你有专属区域、代理地址或企业网关，需要在这里显式填写。
- **认证凭据**：通常是 API Key，也可能是 AccessKey、Secret 或其他配置。
- **模型配置**：声明渠道支持哪些模型，并处理必要的重定向与别名。

## 常见的路由组织方式

### 按供应商拆分

把 OpenAI、Anthropic、Google、Azure 等供应商分别做成独立渠道，最适合看清成本来源和故障边界。

### 按环境拆分

例如开发环境走测试额度，生产环境走正式额度，防止调试流量误伤生产预算。

### 按业务分组拆分

为不同团队或租户配置不同分组，再给每组单独绑定可访问的渠道和模型。

## 失败重试与高可用

当你配置了多个渠道后，可以把它们用于重试和流量转移。实践上建议：

- 核心模型至少准备两个可替代出口。
- 对成本差异较大的模型，先把高性价比渠道放在主路径。
- 对不稳定的上游，避免让其承担唯一出口。

## 模型映射的使用场景

模型映射适合处理下面几类问题：

- 客户端只认统一模型名，但不同供应商上游命名不同。
- 需要按环境或渠道把同一个逻辑模型切到不同上游。
- 需要把兼容格式请求转换到特定供应商的真实模型名。

## 建议的排查顺序

当某个模型调用失败时，优先按以下顺序排查：

1. 渠道认证是否仍然有效。
2. 渠道是否声明了该模型，或模型映射是否配置正确。
3. 上游地址是否被代理、区域配置或企业网络策略拦截。
4. 用户分组、模型限制和速率限制是否挡住了请求。

## 管理建议

- 不要把所有上游都塞进一个大而全的渠道里，后续维护会很难定位问题。
- 把渠道名称写得能反映地区、供应商和用途，而不是只写一个模糊别名。
- 在上线前用价格页和调用日志交叉核对模型是否真的被正确路由。
