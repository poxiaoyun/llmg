# API 密钥与调用

API 密钥是应用接入 LLMG 的最小凭证单位。它决定了谁可以访问网关、可以调用哪些模型，以及消耗如何落到账户和分组里。

## 创建一把可用的密钥

在控制台的“API 密钥”页面，新建密钥时建议同时考虑以下维度：

- **用途**：开发、测试、生产最好分别创建。
- **分组**：把密钥绑定到正确的用户组或业务组，便于限流与隔离。
- **模型白名单**：只放行真正需要的模型，减少误用成本。
- **有效期**：给临时环境和外部合作方设置明确过期时间。

## 建议的密钥划分方式

### 按环境分开

为本地开发、测试环境、生产环境分别创建密钥，便于撤销和审计。

### 按应用分开

不要让多个应用共用一把密钥。独立密钥能让你更快定位异常使用量和上游错误。

### 按团队分开

如果同一套网关服务多个团队，建议用分组、模型限制和额度配额做团队边界。

## 首条调用建议

最稳妥的起点依然是 OpenAI 兼容的 `chat/completions` 接口，因为大多数 SDK、低代码平台和自定义客户端都已经支持这一模式。

```bash
curl http://localhost:3000/v1/chat/completions \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o-mini",
    "messages": [
      {"role": "system", "content": "You are a routing smoke test."},
      {"role": "user", "content": "返回一句简短的成功提示。"}
    ],
    "temperature": 0.2
  }'
```

## 调用后要看什么

- 密钥额度是否被正确扣减。
- 请求是否进入预期的渠道或模型映射。
- 失败时是否能在日志里看到明确的上游错误信息。
- 响应格式是否符合你的 SDK 或客户端预期。

## 轮换与失效处理

### 定期轮换

长期在线的生产密钥建议设置轮换窗口，不要把单把密钥无限期使用下去。

### 临时吊销

一旦怀疑泄露，先在控制台禁用或删除该密钥，再重新发放新密钥，不要等上游账单异常后再处理。

### 回收历史密钥

定期清理已过期、废弃应用和不再使用的测试密钥，避免审计噪音。

## 常见误区

- 只创建一把全局密钥，导致无法区分不同业务的成本和错误。
- 把管理员密钥直接发给应用侧，扩大了配置和数据暴露面。
- 不限制模型，结果用户直接调用了高成本或未验证的模型。
